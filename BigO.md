1. logger_util.py  
Complexidade de Tempo: $O(1)$ - Constante  
As operações neste arquivo incluem capturar a data/hora atual (datetime.now()), formatar uma string e fazer o append (adicionar ao final) num arquivo de texto. Nenhuma destas ações depende do tamanho de uma lista ou entrada externa que possa crescer indefinidamente. Escrever no final de um arquivo é uma operação $O(1)$.

2. notifier.pyComplexidade de Tempo: $O(1)$ - ConstanteA função send_email_notification monta um objeto de e-mail e abre uma conexão SMTP. O tempo que a rede demora para enviar o pacote pode variar devido à latência da internet, mas do ponto de vista do algoritmo (ciclos de CPU), o número de instruções executadas é sempre o mesmo, independentemente do conteúdo do e-mail.

3. verifier.pyComplexidade de Tempo: $O(N)$ - LinearEste é o núcleo do processamento de dados do sistema.Seletores e DOM: O ciclo for target in targets roda um número fixo de vezes (uma constante). Dentro dele, page.locator(target).count() obriga o motor do Chromium a pesquisar a árvore DOM. No pior caso, pesquisar uma árvore não estruturada leva $O(N)$, onde $N$ é o número de nós do HTML.Expressão Regular (Fallback): O comando page.content() extrai todo o HTML gerando uma string gigante de comprimento $M$. O motor de Regex do Python processa a busca com re.search de forma linear sobre a string: $O(M)$.Limpeza Inteligente: Todas as manipulações (rfind, replace, converções) acontecem sobre a string final num_str (que possui apenas alguns caracteres, como "1.500,00"). Sendo um tamanho minúsculo e fixo, o custo é $O(1)$.Resumo: A extração domina o custo com $O(N + M)$. Como nós do DOM e o tamanho do HTML são proporcionais, a complexidade é definida como $O(N)$.

4. monitor.pyComplexidade de Tempo: $O(C \times N)$ - Linear/MultiplicativaAs funções auxiliares _is_valid_variation, _update_monitor_csv e _append_history_csv realizam matemática simples e escrita em arquivo, todas em $O(1)$.A função start_monitoring possui um while True. Este ciclo executa indefinidamente de acordo com o intervalo definido no delay. Se considerarmos $C$ como o número de ciclos (vezes que o loop rodou antes do utilizador fechar o programa), e sabendo que dentro de cada ciclo ele chama o verifier.py que custa $O(N)$, o custo computacional desta função multiplica o número de ciclos pelo tamanho da página.

5. ui.py e main.pyComplexidade de Tempo: $O(1)$ para a Interface, delegando para $O(C \times N)$ na ExecuçãoTanto o main.py procedural quanto o ui.py com o CustomTkinter possuem tempo constante $O(1)$ para construir a interface, validar variáveis com Regex e instanciar os elementos visuais.Quando o utilizador clica em "Confirm & Start Monitoring", a interface cria uma Thread que executa a lógica do monitor.py. Portanto, a interface permanece leve e constante, transferindo o peso algorítmico do laço infinito ($O(C \times N)$) para o processador em segundo plano.Conclusão: Big O Geral do SistemaComplexidade de Tempo Geral: $O(C \times N)$O tempo total gasto pelo processador será proporcional ao tamanho da página web sendo monitorada ($N$) multiplicado pela quantidade de vezes que o sistema for configurado para checar a página ($C$).Complexidade de Espaço (Memória): $O(N)$O pico de uso de memória ram ocorre quando o Playwright carrega a aba do Chromium invisível e quando a função page.content() cria uma cópia inteira do código-fonte na variável html. Ambos os consumos crescem linearmente conforme o tamanho da página web do produto. Arquivos gigantes consumirão mais RAM do que páginas simples.
